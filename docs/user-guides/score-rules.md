# スコアルールの使い方（利用者向け）

vCenter Event Assistant では、取り込んだ **イベント** に **要注目スコア**（0〜100 の数値）を付け、一覧や概要で優先度を判断しやすくします。環境に合わせた調整は **スコアルール**（**設定 → スコアルール**）で行います。本書は開発者向けではなく、スコアの意味・設定・他機能との関係を把握したい運用担当者向けに、現行の挙動をまとめたものです。

---

| あなたの役割 | まず読む節 | 目的 |
|-------------|-----------|------|
| **運用（設定）** — 種別ごとの重要度を調整する | [§4](#4-スコアルールの考え方) → [§7 設定](#7-画面別操作) | 環境に合わせてスコアを上げ下げする |
| **運用（調査）** — 一覧・概要でイベントを追う | [§3](#3-要注目スコアとは) → [§6](#6-他機能との関係) | 表示されている数値の読み方 |
| **アラート連携** — メール通知の条件を決める | [§6](#6-他機能との関係) → [アラート機能の使い方](alerts.md) | スコア変更が通知に与える影響 |

用語が似ているので、次の4つを区別してください。

| 用語 | 意味 |
|------|------|
| **要注目スコア** | イベント1件に付く重要度（0〜100）。**イベント** タブの **スコア** 列や **概要** の一覧で確認する。DB 上は `notable_score`。 |
| **スコアルール** | **設定 → スコアルール** で定義する、イベント種別ごとの **加算値**。サーバーに保存される。 |
| **重大度（severity）** | VMware 由来の分類（error / warning など）。**要注目スコアとは別物** で、スコアルールでは変更しない。 |
| **アラートルール** | **設定 → アラート** の「メールを送る条件」。イベントスコア型は要注目スコアと閾値を比較する（[alerts.md](alerts.md)）。 |

## 2. スコア付けの全体像

1. vCenter から **イベント** を取り込む。
2. アプリが **ベーススコア** を、イベント種別・重大度・メッセージなどから自動算出する（画面では変更不可）。
3. その種別に **スコアルール** があれば **加算** し、結果を **0〜100** に収めて **要注目スコア** として保存する。
4. **イベント** タブ・**概要**・**タイムライン**・**アラート**（イベントスコア型）などが、このスコアを参照する。

```
[イベント取り込み]
        ↓
[ベーススコアの自動算出]
        ↓
[スコアルールの加算]（種別ごと・未設定は +0）
        ↓
[0〜100 に収める] → 要注目スコア（保存）
        ↓
    ┌───┴───┬───────────┬────────────┐
    ↓       ↓           ↓            ↓
[イベント一覧] [概要・ダイジェスト] [タイムライン] [アラート評価]
```

ルールの **追加・変更・削除・インポート** のあとは、**すでに取り込み済みのイベント** に対してもスコアが **再計算** されます（§5）。

## 3. 要注目スコアとは

- **要注目スコア** は、「このイベントをどれだけ注意深く見るか」を表す **0〜100** の整数です。
- **イベント** タブの表では **スコア** 列に表示されます。**概要** の「要注意イベント（上位）」や統計でも同じ値を使います。
- 最終スコアは次のイメージです。

  **要注目スコア = ベーススコア + スコアルールの加算**（結果を 0 未満にはせず、100 を超えないよう収める）

### ベーススコア（自動）

取り込み時および再計算時に、アプリが次のような情報から **ベース** を決めます。

- イベント **種別**（`event_type`）
- VMware の **重大度（severity）**
- **メッセージ** 本文（キーワードなど）

ベースの細かな重み付けや種別リストは **画面では変更できません**。アルゴリズムの詳細は開発者向けの [backend.md](../backend.md) を参照してください。

### スコアルールによる加算

**設定 → スコアルール** で、種別ごとに **加算**（負の数も可）を登録します。ルールが無い種別は加算 **0** です。

### VMware の重大度との違い

イベント行の **重大度** は vCenter 由来の分類です。**要注目スコア** はアプリ独自の運用向け指標であり、数値も一致しません。緊急度の判断には、スコアとあわせて [アラート](alerts.md) のレベル（クリティカル・エラー・警告）も参照してください。

## 4. スコアルールの考え方

**設定 → スコアルール** で管理するのは、次の2項目だけです。

| 項目 | 説明 |
|------|------|
| **イベント種別** | 収集ログ・一覧に出る `event_type` 文字列と **完全一致** させる（例: `vim.event.VmPoweredOnEvent`）。短い別名やワイルドカードは使えない。 |
| **加算** | ベーススコアに足す整数。**負の数** で下げることもできる。 |

- **1 種別につき 1 ルール** です。同じ種別を二重登録することはできません。
- **最終スコアは常に 0〜100** です。加算の結果が 100 を超えても 100、0 未満になっても 0 として保存されます。
- 運用の例: テスト環境では無視したい種別の加算を **負** にする、本番で見逃したくない種別の加算を **大きく正** にする。

イベント種別の意味や対処の目安は、別途 **設定 → イベント種別ガイド** や [event-type-guides/README.md](../event-type-guides/README.md) を参照してください（ガイドはスコア計算には影響しません）。

## 5. 保存と再計算

- スコアルールは **サーバー**（データベース）に保存されます。**設定 → 一般** のタイムゾーンなどとは異なり、**他のブラウザや端末と共有** されます。
- **追加**、一覧での **保存**（加算の変更）、**削除**、**ファイルからインポート** のいずれでも、該当する **取り込み済みイベント** の要注目スコアが **再計算** されます。
- **削除** した種別は加算 0 として再計算され、ベーススコアのみの値に戻ります（確認ダイアログの文言どおり）。
- **これから取り込まれるイベント** も、取り込み時点のルールでスコアが付きます。

再計算はバックグラウンドで一括処理されます。件数が多い環境では、反映に少し時間がかかることがあります。

## 6. 他機能との関係

要注目スコアは複数の画面で使われますが、**表示の切り口や閾値は機能ごとに異なります**。

| 機能 | スコアの使われ方 |
|------|------------------|
| **イベント** タブ | 各行の **スコア** 列。絞り込み・運用メモは [events.md](events.md)。 |
| **概要** | **直近24h** ダッシュボード。統計・上位一覧の詳細は [summary.md](summary.md)。スコアルール変更は次回再取得後に反映。 |
| **ダイジェスト** | 生成レポートの「要注意」系の集計でもスコアを参照（要注意件数はスコア ≥40。詳細は [digests.md](digests.md)）。 |
| **タイムライン** | 期間内イベントを種別ごとに集計し、バケットごとに **max スコア** の高い種別を **アラート** 行として表示。**アラート上位件数**（1〜20）はタイムライン画面で指定（ブラウザに記憶）。**メールは送らない**（[alerts.md](alerts.md) §7）。 |
| **アラート（イベントスコア型）** | 要注目スコアがルールの **閾値以上** かつ評価ウィンドウ内にあると **発火・再通知** の対象。スコアルールを変えると、同じイベントでも通知の出方が変わる（[alerts.md](alerts.md) §3・§5）。 |

スコアルールを変更したあと **メールがすぐ変わらない** 場合は、アラート側の **閾値**・**有効** フラグ・**再通知間隔**・**評価ウィンドウ** を [alerts.md](alerts.md) §8 で切り分けてください。

## 7. 画面別操作

### 設定 → スコアルール

画面の構成は [設定のスコアルール（スクリーンショット）](../images/settings-score-rules.png) を参照してください。

#### エクスポート・インポート

- **ファイルにエクスポート**: 現在のルール一覧を JSON でダウンロードする。
- **ファイルからインポート**: JSON を読み込む。**インポート時のオプション** はインポート時のみ有効（§8）。
- パネル上部の説明どおり、ルールは **サーバー** に保存され、インポート後も **既存イベントの再計算** に反映される。

#### 追加

1. **イベント種別（完全一致）** に `event_type` を入力する。
2. **加算（負数可）** を指定する。
3. **追加** を押す。同じ種別が既にある場合はエラーになる。

#### 一覧

| 列 | 操作 |
|----|------|
| イベント種別 | 参照のみ（変更するには削除して作り直す） |
| 加算 | 数値を編集し **保存** |
| （操作） | **保存** で PATCH、**削除** でルール削除（既存イベントはベースに戻る） |

## 8. JSON バックアップ

アラートルールと同様、環境間の移行やバックアップに JSON を使えます。

### ファイル形式

- 識別子 **`format`**: `"vea-event-score-rules"`（必須）
- **`version`**: 整数（現行は `1`）
- **`rules`**: 配列。各要素は `event_type` と `score_delta`
- ファイル内で **同じ `event_type` が重複** していると読み込みエラーになる

エクスポート時のファイル名の例: `vea-score-rules-2026-05-23.json`

### インポート時のオプション

| オプション | 意味 |
|-----------|------|
| **既存の同一イベント種別を上書き** | オン: ファイルと同じ種別のルールは加算を更新。オフ: 既存はそのまま（新規種別のみ追加）。 |
| **ファイルに含まれないイベント種別のルールを削除** | オン: インポート後、ファイルに無い種別のルールを **削除** する。オフ: ファイル外の既存ルールは残す。 |

### 注意

- **ファイルに含まれないルールを削除** にチェックを入れたうえで、**ルールが空のファイル** をインポートすると、**既存ルールがすべて削除** されます。確認ダイアログの内容をよく読んでから実行してください。
- インポートは **サーバー上のルール定義** を変え、続けて **イベントの再計算** を行います。アラートルール JSON とは **別形式** です（混同しないこと）。

## 9. よくある質問と切り分け

### スコアを上げたのにイベント一覧の数値が変わらない

1. 一覧で **保存** を押したか（加算欄を変えただけでは未保存）。
2. 対象行の **種別** が、ルールの `event_type` と **完全一致** しているか（大文字小文字・プレフィックス `vim.event.` を含む）。
3. 再計算後に **イベント** タブを再読み込みしたか。
4. 加算しても **ベース + 加算** がもともと 100 付近なら、表示は 100 のまま変わらないことがある。

### スコアルールを変えたのにメールが来ない／来方が変わらない

- スコアルールは **要注目スコア** だけを変えます。**メール** は **設定 → アラート** の **イベントスコア型** ルールが有効で、閾値・評価ウィンドウ・再通知間隔を満たしたときに送られます。[alerts.md](alerts.md) §8 を参照してください。

### 概要の「24h 要注意」と一覧のスコアの関係

- 詳細は [summary.md](summary.md) §3・§4・§8 を参照。
- **24h 要注意（スコア≥40）** と **要注意イベント（上位）** は切り口が異なります（40 固定の件数 vs 最小スコアで絞った最大 10 行）。

### 同じイベント種別を二つ登録できない

- 仕様どおり **1 種別 1 ルール** です。加算だけ変える場合は既存行を **保存** で更新してください。

### タイムラインにアラート行が出るがメールが来ない

- 正常です。タイムラインの **アラート** 表示は調査用の集計であり、[アラートルール](alerts.md) による通知とは別経路です（§6）。

### イベント種別ガイドとスコアルールの違い

| | スコアルール | イベント種別ガイド |
|--|-------------|-------------------|
| 目的 | 数値（要注目スコア）の調整 | 意味・原因・対処の説明表示 |
| 保存先 | サーバー | サーバー |
| 一覧への影響 | **スコア** 列 | **ガイド** 列・行の強調（対処が必要） |
