# アラート機能の使い方（利用者向け）

vCenter Event Assistant の **アラート** は、主に **メール通知** と **画面での確認** の2系統があります。本書は開発者向けではなく、ルール設定・通知対応・障害調査の担当者向けに、現行の挙動をまとめたものです。

---

| あなたの役割 | まず読む節 | 目的 |
|-------------|-----------|------|
| **運用（設定）** — ルールを作る・変更する | [§4](#4-アラートルールの種類と重大度レベル) → [§7 設定](#7-画面別操作) | いつメールが飛ぶかを定義する |
| **運用（受信）** — メールを受け取って対応する | [§6](#6-メール通知の見方と対応の目安) | 件名・本文の読み方と優先度 |
| **障害調査** — 履歴やタイムラインで追う | [§7 通知履歴・タイムライン](#7-画面別操作) → [§8](#8-よくある質問と切り分け) | 発火・回復の時系列と送信結果 |

用語が似ているので、次の4つを区別してください。

| 用語 | 意味 |
|------|------|
| **要注目スコア** | イベント一覧に表示される重要度（数値）。スコアルールで調整される。 |
| **アラートルール** | **設定 → アラート** で定義する「メールを送る条件」。 |
| **通知履歴** | メインタブ **通知履歴** に記録される、発火・回復と送信結果の一覧。 |
| **タイムライン上のアラート** | **タイムライン** タブで期間内の高スコアイベントを可視化したもの。**メールは送らない**。 |

## 2. アラート機能の全体像

1. vCenter から **イベント** と **ホストメトリクス**（CPU/メモリなど）を定期的に取り込む。
2. **スコアルール** で各イベントに **要注目スコア** を付ける。
3. **アラートルール** が一定間隔で条件を評価し、条件を満たすと **発火** し、回復すると **回復** として記録する。
4. 発火・回復のたびに **通知履歴** に1行追加される。SMTP が設定されていれば **メール** も送られる。
5. **タイムライン** では、調査用に同じ期間のイベント・メトリクス・高スコアイベントをまとめて表示する（通知とは別経路）。

```
[イベント/メトリクス取り込み]
        ↓
[スコアルール] → 要注目スコア
        ↓
[アラートルール] → 定期評価 → 発火 / 回復
        ↓                    ↓
[通知履歴タブ]          [メール（SMTP 設定時）]
        ↓
[タイムライン] ← 調査用の可視化（メールなし）
```

---

## 3. 要注目スコアとスコアルール（通知の前提）

- **要注目スコア** は、イベントが「どれだけ注意すべきか」を表す数値です。**イベント** タブの一覧や **概要** の要注目表示で確認できます。
- **スコアルール**（**設定 → スコアルール**）で、イベント種別ごとにスコアの加減を設定できます。環境に合わせて重要度を上げると、アラート判定の材料にもなります。
- イベント行の **重大度（severity）** は VMware 由来の分類です。**アラートのレベル（クリティカル・エラー・警告）とは別物** です。運用上の緊急度はアラートルール側のレベルを参照してください。

**イベントスコア型のアラート** は、「要注目スコアがルールの閾値以上のイベントがあるか」で判定します。スコアルールを変えると、同じイベントでもスコアが変わり、通知の出方が変わることがあります。

---

## 4. アラートルールの種類と重大度レベル

**設定 → アラート** でルールを管理します。有効にしたルールだけが評価されます。

### ルールの種類（タイプ）

| 画面での名称 | 判定内容 | 対象の例（通知の「対象」列） |
|-------------|---------|---------------------------|
| **イベントスコア** | 要注目スコアが **閾値以上** のイベントがある | イベント種別名（例: `vim.event.UserLoginSessionEvent`） |
| **メトリクス閾値** | 指定メトリクスが **閾値以上** のホストがある | ホストの識別子（内部 ID。メール本文の Details にホスト名が出る場合あり） |

- 新規作成時、イベントスコア型は **クールダウン（分）** が既定 10 分で付きます（行の編集から変更可能）。
- メトリクス型は **メトリクスキー** を、グラフタブやキー一覧と **完全一致** させてください（例: CPU 利用率 `host.cpu.usage_pct`）。キーが DB のサンプルと一致しないと発火しません。

### 重大度レベル（クリティカル・エラー・警告）

ルールごとに1つ選びます。メールの件名と **通知履歴** の「レベル」列に反映されます。

| レベル | 運用上の目安 |
|--------|-------------|
| **クリティカル** | サービス影響の可能性が高く、**すぐ**対処が必要 |
| **エラー** | サービス影響の可能性が高く、**対処が必要** |
| **警告** | 直接的なサービス影響は想定しにくく、**状況の検討**が必要 |

新規ルールの既定レベルは **警告** です。重要なルールは運用方針に合わせて引き上げてください。

### ルールのバックアップ

- **ファイルにエクスポート** / **ファイルからインポート** で JSON のやり取りができます。
- インポート時の「ファイルに含まれないルールを削除」に注意してください。誤操作で既存ルールが消える可能性があります。

## 5. 発火・回復・再通知・クールダウン

アラートは **発火（firing）** と **回復（resolved）** の2状態で管理されます。メールと通知履歴は、**発火または回復に遷移したとき**、および（イベントスコア型のみ）**条件を満たすより新しいイベントが現れたとき** に記録されます。

### 共通

- **発火**: 条件を満たし、以前は未発火または回復済みだったとき → **発火** の通知が1回。
- **回復**: 条件を満たさなくなったとき → **回復** の通知が1回。

### イベントスコア型

- **判定対象**: イベント一覧と同じ **要注目スコア** が閾値以上で、かつ **直近一定時間内**（評価ウィンドウ）に発生したイベント。ウィンドウの長さはサーバー管理者が設定します（ルールごとの設定項目ではありません）。
- **発火・再通知**: ウィンドウ内に閾値以上のイベントがあるとき、初回または回復後は **発火** 通知。すでに発火中でも、**より新しい** 閾値以上イベントが来ると **再度発火** 通知します（メールの対象・通知履歴の「対象」は **イベント種別名**）。
- **継続（再通知なし）**: 発火中で、新しい閾値以上イベントが来ない間は **追加のメールは送りません**。
- **回復**: ウィンドウ内に閾値以上のイベントがなくなり、最後の発火から **クールダウン（分）** 経過後に **回復** 通知（既定 10 分。ルール編集で変更可）。

### メトリクス閾値型

- **ホストごと** に状態を持ちます。ホスト A が閾値超え、ホスト B が正常、といった組み合わせが可能です。
- あるホストで閾値超え → そのホスト向けに発火。値が閾値未満に下がる → そのホスト向けに回復。
- 閾値超えが **続いている間は再通知しません**（ホストごとの状態更新のみ）。回復後に再び超えたときだけ再発火します。

### 評価のタイミング

- サーバーはおおむね **1分ごと**（管理者が変更可能）に有効ルールを評価します。メールが届くまでに最大でその程度の遅れがあり得ます。

## 6. メール通知の見方と対応の目安

SMTP と送信先がサーバー側で設定されているときだけ、メールが配信されます（未設定でも **通知履歴** には記録される場合があります）。

### 件名の形式（既定テンプレート）

| 状態 | 件名の例 |
|------|---------|
| 発火 | `[FIRING][warning] ルール名 - vim.event.ExampleEvent` |
| 回復 | `[RESOLVED][error] ルール名 - host-123` |

- 角括弧内の `critical` / `error` / `warning` はレベルです。本文にも日本語ラベル（クリティカル等）が含まれます。
- **Resource**（対象）行は、イベント型では **イベント種別名**、メトリクス型では **ホスト識別子** です。
- **Details** にスコアやメトリクス値、メッセージの要約が入ります。

### 受信時の対応の目安

1. 件名の **レベル** で優先度を判断する（§4 の表）。
2. **ルール名** でどの条件に引っかかったかを把握する。
3. **Resource** でイベント種別かホストかを切り分ける。
4. 必要ならアプリの **イベント** タブや **グラフ** タブ、**タイムライン** で前後を確認する。

## 7. 画面別操作

### 設定 → アラート

- **新規ルール追加**: 名前、タイプ（イベントスコア / メトリクス閾値）、レベル、閾値（およびメトリクスキー）を指定して保存。
- 一覧の **有効** チェックでルールの ON/OFF。
- 行の **▸** で名前・閾値・クールダウン（またはメトリクスキー）を編集。**タイプは後から変更できません**（作り直しが必要）。
- レベルは一覧のプルダウンから即時変更できます。

### 通知履歴（メインタブ）

| 列 | 内容 |
|----|------|
| 日時 | 通知が記録された時刻（タイムゾーンは **設定 → 一般** に従う） |
| ルール名 | 発火したルール |
| レベル | クリティカル / エラー / 警告 |
| 状態 | **発火中** または **回復済み** |
| 対象 | イベント種別名、またはホスト識別子 |
| 結果 | **成功**（送信処理成功）または **失敗**（SMTP エラー等。マウスオーバーで詳細） |

- **一覧を更新** で再読み込み。30秒ごとに自動更新されます。
- 履歴が空でも、ルールが無効・閾値未満・評価待ちの可能性があります（§8）。

### タイムライン（メインタブ）

- **メール通知とは無関係** に、指定期間のイベント・メトリクス・高スコアイベントをまとめて表示します。
- **要注目の閾値**（タイムライン用）と **アラート上位件数**（1〜20、バケットごとに表示する種別数）を指定して **タイムラインを生成** します。
- **表示順**（昇順 / 降順）を切り替えられます。設定はブラウザに記憶されます。
- タイムライン上の **アラート** 行は、期間内でスコアの高いイベント種別の集計表示です。§4 のアラートルールでメールが飛ぶかどうかとは別設定です。

## 8. よくある質問と切り分け

### イベント一覧に高スコアがあるのに、メールも通知履歴もない

1. **設定 → アラート** で、該当条件のルールが **有効** か。
2. ルールの **閾値** が、一覧の要注目スコア **以下** になっているか（「60以上」ならスコア 59 では発火しない）。
3. イベントが **評価ウィンドウ（直近一定時間）の外** にあるだけでは発火しない（§5）。ウィンドウの長さは管理者に確認する。
4. **メトリクス型** で、すでに発火中かつ値が閾値超えのままでは **再通知されない**（§5）。**イベントスコア型** は、より新しい閾値以上イベントがあると再通知される。
5. メトリクス型なら **メトリクスキー** がグラフと一致しているか、データが取り込まれているか。

### 通知履歴は「成功」だがメールが届かない

- 履歴の成功は **アプリが送信処理を完了したこと** を示します。受信側の迷惑メール、転送設定、SMTP の From/To は **サーバー管理者** が確認します（[backend-operations.md](../backend-operations.md) の SMTP 節）。

### 通知履歴が「失敗」になる

- 結果列にマウスオーバーするとエラー概要が出ます。SMTP 認証・TLS・宛先の設定不備が多いです。管理者に連絡してください。

### 同じイベント種別で何度もメールが来る

- **イベントスコア型**: 発火中でも、ウィンドウ内に **より新しい** 閾値以上イベントがあると **再び発火** 通知が送られます（意図した挙動）。
- **回復** のあと、再び閾値以上のイベントが発生すると **再度発火** します。
- **メトリクス型**: 閾値超えが続く間は再通知しません。連続メールは別ルール・別ホスト・回復と再発火の繰り返しを通知履歴の **状態** 列で確認してください。

### タイムラインにアラートが出るがメールが来ない

- 正常な場合があります。タイムラインは調査用表示であり、**アラートルールが有効で条件を満たし、発火遷移したとき** にだけメール用の通知が記録されます。

### 古いイベントで突然アラートが出た

- イベントスコア型は **評価ウィンドウ内** のうち、閾値以上で **最も新しい1件** だけを見ます。ウィンドウ外の古いイベントだけが閾値を超えている場合は通常発火しません。ウィンドウ内の古いイベントで発火した場合は、ルール閾値・スコアルール・ウィンドウ設定（管理者）の見直しを検討してください。
