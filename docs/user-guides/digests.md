# ダイジェスト機能の使い方（利用者向け）

<!-- vea-tab-help: digests -->

vCenter Event Assistant の **ダイジェスト** は、指定期間に取り込まれた **イベント** と **ホストメトリクス** を集計し、**Markdown レポート** としてサーバーに保存する機能です。**ダイジェスト** タブで過去分を閲覧・ダウンロードできます。環境によっては **LLM 要約** が本文末尾に付きます。本書は開発者向けではなく、レポートの読み方と画面操作を把握したい運用担当者向けに、現行の挙動をまとめたものです。

| **運用（比較）** — 日次と週次の傾向を見る | [§3](#3-ダイジェストとは) → [§7](#7-画面別操作) | 種別（daily / weekly / monthly）の切り替え |
| **管理者** — 生成スケジュールや LLM を設定する | [§5](#5-生成の仕組み管理者向け概要) → [getting-started.md](../getting-started.md) | いつ・どう作られるかの前提 |

用語の区別:

| 用語 | 意味 |
|------|------|
| **ダイジェスト** | 1 期間分の集計レポート（Markdown）。**ダイジェスト** タブの一覧から選んで表示する。 |
| **種別（kind）** | `daily`（日次）・`weekly`（週次）・`monthly`（月次）。一覧のフィルタボタン名と一致する。 |
| **集計期間** | レポートの対象となる時間範囲。DB 上は UTC の **半開区間** `[開始, 終了)`（終了時刻は含まない）。画面の日時表示は **設定 → 一般** のタイムゾーンに従う。 |
| **テンプレート本文** | サーバーが Jinja テンプレートから生成する固定構成の Markdown（表・箇条書き）。 |
| **LLM 要約** | テンプレート本文の **後ろ** に追記される `## LLM 要約` セクション（任意）。 |

## 2. ダイジェスト機能の全体像

1. バックグラウンドの **定期ジョブ** または管理者による **API 手動実行** で、対象期間のデータを集計する。
2. **テンプレート** から Markdown 本文（イベント統計・要注意イベント・ホスト CPU/メモリ上位など）を生成する。
3. **LLM が設定されている** 場合は、集計結果を入力に要約ブロックを追記する（失敗時はテンプレートのみ保存し、警告文を付けることがある）。
4. 結果を **サーバー** に保存する（一覧の各行が 1 レポート）。
5. 利用者は **ダイジェスト** タブで一覧・本文表示・Markdown ダウンロードを行う（**画面から新規生成はできない**）。

```
[vCenter からイベント・メトリクス取り込み]
        ↓
[指定期間で集計] → DigestContext（件数・種別・スコア・ホスト上位など）
        ↓
[テンプレートで Markdown 生成]
        ↓
[（任意）LLM で「## LLM 要約」追記]
        ↓
[サーバーに保存] → ダイジェストタブで閲覧・ダウンロード
```

## 3. ダイジェストとは

### 種別（daily / weekly / monthly）

| 種別 | おおむねの対象期間（期間省略の自動実行時） |
|------|------------------------------------------|
| **daily** | **表示タイムゾーン**（サーバー設定 `DIGEST_DISPLAY_TIMEZONE`）における **直前の暦日**（その日 0:00 〜 翌日 0:00、半開区間で DB には UTC 換算） |
| **weekly** | 同 TZ で **日曜 0:00 始まり** の直前 **7 日間**（完了した暦週） |
| **monthly** | 同 TZ で **直前の暦月**（1 日 0:00 〜 翌月 1 日 0:00） |

利用者が一覧で見る **期間の始まり** は、種別と `period_start` に基づき、**設定 → 一般** のタイムゾーンで日付表示されます。詳細な境界の計算はサーバー管理者向けの [backend.md](../backend.md) および `.env.example` を参照してください。

### 保存と重複

- 同じ期間・種別で **何度実行しても**、実行のたびに **新しい行** が追加されます（上書きではありません）。運用では最新の `作成` 日時を目安に読むとよいです。
- 一覧の **status** が `error` の行は、テンプレート生成に失敗したレポートです（§5）。

## 4. 本文の見方

テンプレート（同梱の `digest.md.j2`）では、おおむね次の順で並びます。

### 先頭のサマリー

- **集計期間**（UTC で集計、表示はタイムゾーン表記）
- **登録 vCenter 数**
- **イベント総数**
- **要注意（notable_score ≥ 40）件数** — 概要タブの「24h 要注意（スコア≥40）」と **同じ閾値** の件数です（期間内の全イベントに対するカウント）。

### 上位イベント種別（件数順）

期間内で **件数の多いイベント種別** と、その種別の **最大スコア**（max notable）の表です。どの種別が多発しているかの把握に使います。

### 要注意イベント（スコア上位・種別集約）

**要注目スコア** の高いイベントを **種別（`event_type`）ごとに 1 行** にまとめた箇条書きです。

- **score** … その種別の期間内 **最大** スコア
- **N 回発生** … 同種別の出現回数
- 時刻レンジ … 初回〜最終の `occurred_at`
- **entity** … 同一エンティティのみのとき名前、複数 VM 等では `—（複数）`

テンプレート上は **最大 20 グループ** まで表示し、それ以上は省略行が付きます。集約前にサーバーが参照するグループ数には別上限があります（利用者は省略表示が出たら、**イベント** タブで詳細を追う想定）。

スコアの付け方・調整は [score-rules.md](score-rules.md) を参照してください。

### ホスト CPU / メモリ利用率（ピーク上位）

期間内サンプルの **ピークが高いホスト** を、vCenter ラベル・ホスト名・利用率・サンプル時刻付きで示します。グラフタブの時系列とあわせて負荷の偏りを確認できます。

### LLM 要約（任意）

- サーバーに **LLM 要約あり（モデル名）** と表示がある、または本文に `## LLM 要約` がある場合に表示されます。
- 要約は **テンプレートに既にある数値の言い換えだけ** ではなく、運用上の **読み方・確認の優先度** などの補足を意図しています。
- **LLM が未設定**、または保存時に `llm_model` が記録されていない場合、画面は **`## LLM 要約` ブロックを表示しません**（テンプレート部分のみ）。
- LLM 呼び出しに失敗したがテンプレートは保存された場合、画面上部に **警告文**（`error_message`）が出ることがあります。本文はテンプレート中心で読んでください。

## 5. 生成の仕組み（管理者向け概要）

利用者向け画面では **新規生成ボタンはありません**。レポートは次のいずれかでサーバーに作られます。

| 方法 | 誰が行うか | 備考 |
|------|-----------|------|
| **定期スケジュール** | サーバー（APScheduler） | `DIGEST_DAILY_*` / `DIGEST_WEEKLY_*` / `DIGEST_MONTHLY_*` など（`.env`）。有効化と cron は管理者が設定。 |
| **手動 API** | 管理者・自動化 | `POST /api/digests/run`。JSON で `kind` や `from_time` / `to_time` を指定可能。 |

- **LLM_DIGEST_API_KEY** が空のときは **テンプレートのみ** で保存し、外部 LLM は呼びません（ローカル API を使う場合は `.env.example` のダミーキー例を参照）。
- テンプレートファイルの差し替え・週次/月次専用テンプレートも **環境変数** で行います（[getting-started.md](../getting-started.md)、[backend.md](../backend.md)）。

## 6. 他機能との関係

| 機能 | ダイジェストとの関係 |
|------|---------------------|
| **概要** | リアルタイムに近い **直近24h** ダッシュボード（[summary.md](summary.md)）。**ダイジェスト** は **過去の固定期間** のスナップショット。 |
| **イベント** | 個別行の詳細・フィルタ（[events.md](events.md)）。**ダイジェスト** の種別集約はここで深掘りする。 |
| **グラフ** | メトリクス時系列（[graph.md](graph.md)）。ダイジェストのホスト表は期間内ピークの抜粋。 |
| **スコアルール** | 要注目スコアを変えると、**次回以降に生成される** ダイジェストのスコア・要注意件数に反映される（既存保存分は再生成しない）。 |
| **アラート** | メール通知用。**ダイジェスト** は通知を送らない（[alerts.md](alerts.md)）。 |
| **チャット** | 任意期間を対話で調査（[chat.md](chat.md)）。**ダイジェスト** は定型レポートの保存・閲覧。 |

## 7. 画面別操作

### ダイジェスト（メインタブ）

#### 一覧まわり

- **一覧を更新** … サーバーから最新一覧を再取得する。
- **種別フィルタ**（`all` / `daily` / `weekly` / `monthly`）… 表示するレポートを絞る。切り替えるとページは先頭に戻る。
- **全 N 件** … フィルタ後の総件数。
- 左の一覧 … 各行に **kind**・**期間の開始日**・**status**・**作成** 日時。行をクリックして選択する。
- **前へ / 次へ** … 50 件ずつのページング。

#### 本文（右ペイン）

- **集計期間 · 作成** … 選択中レポートの期間と保存日時。
- **Markdown をダウンロード** … 画面表示と同じ本文（LLM の表示ルールを含む）を `.md` ファイルで保存する。
- **LLM 要約あり（モデル名）** … 要約が成功して記録された場合のみ。
- 警告・`error` 状態 … §4・§5 を参照。

一覧が空のときのメッセージ例: 「保存済みのダイジェストはありません」— 定期ジョブが無効・まだ初回実行前・期間にデータが無いなどが考えられます。管理者に生成設定を確認してください。

## 8. よくある質問と切り分け

### ダイジェストタブに何も出ない

1. サーバーで **定期ダイジェスト** が有効か、少なくとも一度 **手動実行** されたか（管理者）。
2. 対象期間に **イベントが取り込まれている** か（**イベント** タブ・収集設定）。
3. 種別フィルタが `daily` などに絞られていて、該当種別の保存がまだ無いか。

### LLM 要約が付かない

- **正常な場合あり**: API キー未設定時はテンプレートのみ（§5）。
- 本文はあるが要約だけ無い … 保存行に `llm_model` が無い、または LLM 失敗で警告のみ。管理者はサーバーログと `LLM_DIGEST_*` 設定を確認。

### 要注意件数が 0 なのに「要注意イベント」に行がある

- サマリーの **要注意（≥40）** は **40 以上** の件数。
- 下の箇条書きは **スコア 1 以上** の上位種別も含めて集約しています（既定の下限はサーバー側）。スコア 40 未満の事象が要約の「補足」として LLM に載ることもあります。

### 概要の数字とダイジェストが一致しない

- **概要** は直近 24 時間など **別ウィンドウ**。**ダイジェスト** は **日次・週次・月次の暦境界** のスナップショット。比較するときは **同じ期間** を意識してください。

### 同じ日付のダイジェストが複数ある

- 再実行のたびに **新規保存** されます。最新の **作成** 時刻の行を参照するか、運用で古い行を整理する方針を決めてください（画面からの一括削除機能はありません）。

### スコアルールを変えたのに古いダイジェストのスコアが変わらない

- 保存済み Markdown は **生成時点のスナップショット** です。反映後は **新しく生成された** ダイジェストを読んでください。

## 付録: アプリ内ヘルプとの関係

メイン画面の「使い方を表示」には、ダイジェストの短い説明があります。**画面から期間を指定してその場で生成する** 機能は現状ありません。閲覧・ダウンロードと、サーバーに保存されたレポートの読み方は **本ガイド** を参照してください。
