# WEB 調査付記機能の設計判断（2026-07-08 grilling）

障害やその予兆が疑われるイベントについて、原因・解決方法を WEB で調査し、
チャット応答とダイジェストに付記する機能。`/grill-me` セッションで確定した前提と設計判断のまとめ。

機能は 2 モード構成:

- **モード A: 事前調査（event_type 起点）** — バックグラウンドで高スコア event_type を調査・キャッシュし、
  チャット/日次ダイジェストに定型ブロックで添付する。
- **モード B: ユーザー起点の同期調査（チャットのみ）** — ユーザーが説明する障害内容を起点に、
  チャットの応答生成中に WEB 検索して調査結果を返す。ユーザーが有効/無効を指定できる。

## 確定した前提・設計判断

| # | 論点 | 決定 | 理由・補足 |
|---|------|------|-----------|
| 1 | ネットワーク前提 | **環境による（両対応必須）** | 閉域網・プロキシ環境で機能が静かに無効化されるだけで壊れない設計が必須。検索 API キー未設定 = 機能 OFF |
| 2 | トリガー判定 | **イベントスコア閾値** | 既存の notable スコア（`score_event`）を流用。新規判定器は作らない。閾値は Settings 化 |
| 3 | 調査手段 | **検索 API + LLM 要約** | LLM プロバイダのネイティブ検索には依存しない（openai_compatible / copilot_cli で使えないため） |
| 4 | 検索 API | **`SearchProvider` 抽象化 + まず 1 実装** | 将来の社内検索接続も同じ口に。初回実装は未確定（推奨: Tavily。LangChain 統合・LLM 向け要約済み結果） |
| 5 | 匿名化整合 | **検索クエリも匿名化必須** | クエリは event_type・エラーコード等の汎用語のみ。ホスト名・IP・vCenter 名等の固有情報は一切含めない |
| 6 | 実行タイミング | **取り込み/評価とは独立の事前調査** | チャット/ダイジェスト生成時は保存済み結果を添付するだけ。応答遅延ゼロ |
| 7 | 実行位置 | **専用スケジューラジョブ** | 「高スコアかつ未調査の event_type」を DB から拾って調査。取り込みパイプラインに外部 I/O を混ぜない（H-1/H-3 の教訓） |
| 8 | 要約 LLM | **digest プロファイル流用** | バッチ・非対話で性質が近い。設定項目を増やさない。**copilot_cli 選択時は本機能を無効化**（LangChain 非対応のため） |
| 9 | キャッシュ | **event_type 単位で DB 保存 + TTL** | 新テーブル（event_type, 要約, 参照 URL 群, 調査日時, ステータス）。閉域移行後も過去結果を参照可能 |
| 10 | TTL | **成功 90 日 / 成果なし 30 日**（両方 Settings 化） | 「有用な情報なし」もネガティブキャッシュとして行を保存し、毎サイクルの再検索を防ぐ |
| 11 | レート予算 | **1 ジョブサイクル当たり上限 N 件**（既定 5 程度、Settings 化） | スコア順に消化し、残りは次サイクルへ自然に繰り越し。障害ストーム時の課金暴走を防ぐ |
| 12 | 信頼性担保 | **出典 URL 必須 + 免責表示** | 検索結果に実在する URL のみ引用可。「自動調査結果・必ず原典確認」ラベルを UI/ダイジェストに常記 |
| 13 | チャット付記方式 | **応答ストリーム完了後に定型ブロックを連結** | LLM プロンプトには混ぜない。サーバ側が DB の調査結果をそのまま「関連情報」ブロックとして追記 → URL 改変・捏造の余地を排除 |
| 14 | 添付上限 | **スコア上位 N 件（既定 3、Settings 化）** | 対象期間/コンテキスト内の高スコア event_type 上位のみ。トークン・可読性予算を守る |
| 15 | 対象ダイジェスト | **日次のみ** | 「今起きている障害の対処情報」は即応性が価値。週次/月次（振り返り用途)には付記しない |
| 16 | 初回スコープ | **チャット + ダイジェストのみ** | アラートメール添付・イベント一覧 UI 表示は後続フェーズ（キャッシュ設計は流用可能な形にしておく） |

### モード B（ユーザー起点の同期調査）— 2026-07-08 追加 grilling

| # | 論点 | 決定 | 理由・補足 |
|---|------|------|-----------|
| 17 | 発動方式 | **トグル ON のときのみ、LLM がツール（function calling）として検索** | 明示性（ユーザーが許可した会話でしか外部検索しない）と自然さ（検索が必要な質問のときだけ発行）の両立 |
| 18 | クエリ匿名化 | **検索クエリも `LlmAnonymizer` で洗浄してから送信** | ユーザー自由記述にはホスト名・IP 等が含まれうる。固有語除去で検索精度が落ちる可能性は許容 |
| 19 | トグル粒度 | **メッセージ単位の指定（UI チェックボックス等） + サーバ設定の全体 ON/OFF** | 実効条件は AND: サーバ設定 ON かつ 検索 API キー設定済み かつ 当該メッセージで有効指定。バックエンドにユーザー概念がないため per-user 設定は持たない |
| 20 | トグルの範囲 | **「同期検索の許可」と「事前調査キャッシュの表示」は別トグル** | 「外部送信しない」と「調査情報を見せない」は独立した意図。検索 OFF でも取得済みキャッシュの添付は表示トグルに従う |

モード B の設計上の含意:

- **#13（定型ブロック連結）との整合**: 同期検索では検索結果を LLM コンテキストに入れざるを得ない
  （ツール応答として返すため）。出典担保は「本文中の言及は許容しつつ、**正式な出典ブロックはサーバ側が
  実際のツール実行結果（実在 URL）から機械的に生成して末尾に連結**」で維持する。LLM が本文中に書いた
  URL は信頼せず、出典ブロックのみを一次情報とする旨を免責表示に含める。
- **ツール実行ループ**: 既存のチャットは単発ストリーミング（`stream_chat_to_text`）。モード B は
  「LLM → tool_call → 検索実行 → 結果を戻して継続生成」のループが必要で、チャット配管の変更としては
  本機能で最も大きい。copilot_cli 分岐と openai_compatible の非対応モデルでは**モード B を無効化**
  （UI 上もトグルを出さない/無効表示）。
- **レイテンシ**: 検索発動時は応答が数秒〜数十秒伸びる。フロントは「WEB 調査中…」の進行表示を出す。
  検索タイムアウト（Settings 化、例 15 秒）超過時は検索なしで応答を継続する（失敗分離）。
- **モード A との合流**: 同期検索の結果も event_type が特定できる場合はモード A と同じキャッシュテーブルに
  保存してよい（由来カラムで区別）。ただし初回リリースでは書き込みはモード A のみに限定してよい（設計だけ許す）。

## grilling 中に確定させた設計上の含意

- **調査結果はユーザー質問にもイベント個体にも依存しない**（クエリが event_type 単位のため）。
  これが「事前調査 + キャッシュ」方式を成立させる根拠。チャット側は同期検索を一切しない。
- チャット付記（#13）はフロントの表示にも影響する: ストリーム本文とは別のメタデータ
  （または本文末尾の定型 Markdown ブロック）として返す。既存の `stream_chat_to_text` 経路に
  「ストリーム完了後の追記」を挿す位置の設計が実装時の主要ポイント。
- ダイジェスト付記は `digest_markdown` に「関連調査情報」セクションを追加する形。
  LLM 追記（`digest_llm`）とは独立なので、`ok_llm_failed` 時でも調査付記は成立する。
- 検索失敗・LLM 要約失敗は調査ジョブ内で完結させ、チャット/ダイジェスト生成には影響させない
  （L-1 / R-2 と同じ失敗分離ポリシー）。
