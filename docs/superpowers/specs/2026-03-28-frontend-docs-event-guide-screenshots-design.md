# 設計: フロントエンドドキュメント用スクリーンショット（イベントガイド展開・イベント種別ガイド一覧）

## 概要

`docs/frontend.md` に、**イベントタブでガイドを展開した状態**と、**設定のイベント種別ガイド一覧にフォーカスした状態**のキャプチャを追加する。撮影方式は **案1（固定ビューポート + `fullPage: false`）** に統一し、**出力 PNG の横幅・高さ（ピクセル）を既存ドキュメント用キャプチャと同一**にする。

**一覧が縦に長い**場合は画面全体ではなく、**一覧エリアがビューポート内に収まるようスクロール位置を調整**し、**ビューポート範囲のみ**を保存する（長尺のフルページ PNG は作らない）。

---

## 目的

- 利用者が **イベント一覧でのガイドの見え方**と **設定画面のイベント種別ガイド一覧**を、他セクションと **同じ画像サイズ感**で把握できるようにする。
- スクリーンショット取得を **Playwright の既存フロー**（`frontend/e2e/screenshots.spec.ts` と `uv run scripts/capture_ui_screenshots.py`）に組み込み、手動切り出しに依存しない。

## 非目的

- リポジトリ外ホスティング専用の別解像度セット。
- 画像の縮小・拡大による **事後リサイズ**で寸法を合わせること（文字の可読性を損なうため本設計では採用しない）。

---

## 基準寸法（幅・高さ）

既存の `docs/images/*.png` は取得環境により数ピクセル差があり得るため、次で **公式の基準 `W×H`** を定める。

1. **基準ファイル**: 既存ドキュメント用 PNG のうち **`events.png` を基準**とする（イベントタブの代表キャプチャであり、本変更と同じ文脈）。
2. **初回または CI 前**: `events.png` のピクセル幅・高さを計測し、その値を **`DOC_SCREENSHOT_WIDTH` / `DOC_SCREENSHOT_HEIGHT`**（定数名は実装時に `screenshots.spec.ts` 内の定数でよい）として固定する。
3. **新規2枚**（後述のファイル名）も **同一 `W×H`** で保存する。

**注意**: 基準を変える場合（ビューポート変更など）は **`events.png` を含む全ドキュメント用 PNG を同じ手順で再取得**し、寸法の一貫性を保つ。

---

## 撮影方式（案1の確定内容）

| 項目 | 内容 |
|------|------|
| ビューポート | `page.setViewportSize({ width: W, height: H })` で **基準と同一** |
| キャプチャ | `page.screenshot({ path, fullPage: false })`（**ビューポートのみ**） |
| レイアウト調整 | 撮影直前に、見せたい領域（イベント行＋展開ガイド／一覧の先頭付近など）が **ビューポート内に収まる**よう `scrollIntoView` 等で調整する |

**イベントタブ・ガイド展開**

- 「イベント」タブを開き、**ガイド列の `表示` を展開**した状態（ユーザー確認済みの **A**）で撮影する。
- 表の行高が大きい場合も **ビューポート `H` を超えない**よう、対象行が中央付近に来るようスクロールする（全文が収まらない場合は **ビューポート内で分かる範囲を優先**し、仕様として「代表スクリーン」であることを `docs/frontend.md` に一文記載してよい）。

**設定・イベント種別ガイド一覧**

- 「設定」→ **イベント種別ガイド**（実装済みのパネル）を表示する。
- 一覧が長い場合は **一覧の先頭（または説明文＋一覧先頭）がビューポートに収まる**位置にスクロールし、**一覧全体のフルページ**は撮らない。

---

## 追加する成果物

| 種別 | 内容 |
|------|------|
| PNG | 例: `docs/images/events-event-type-guide-expanded.png`（イベント・ガイド展開） |
| PNG | 例: `docs/images/settings-event-type-guides-list.png`（設定・一覧フォーカス） |
| Markdown | `docs/frontend.md` に各画像と短い説明を追加 |
| Markdown | `docs/development.md` の `docs/images` 対応表にファイル名と説明を追記 |
| E2E | `frontend/e2e/screenshots.spec.ts` に上記2枚の取得ステップを追加（既存テストと同一の基準寸法） |

ファイル名は実装時に上記例から変更してよいが、**`development.md` と `frontend.md` とテスト内のパスを一致**させる。

---

## テストデータ前提

- **イベントタブ**: 少なくとも **1件**、**`type_guide` 付き**のイベントが表示される状態が望ましい（空だと展開 UI が撮れない）。E2E の DB シード／既存のメモリ DB 初期データで満たせるかを実装時に確認し、不足なら **スクリーンショット専用のシード**または **取得スキップ条件**を設計する（本設計では「必ず取得」を推奨）。
- **設定・一覧**: ガイドが **0件**のときは一覧が薄くなる。可能なら **複数件のダミー**、または **1件以上**が載る状態で撮る。

---

## エラー処理・保守

- セレクタ変更で取得に失敗した場合は E2E が失敗し、PNG が更新されない。**主要画面の可視性**は `expect` で既存と同様に担保する。
- `capture_ui_screenshots.py` の挙動（既存サーバー向け `--existing` 等）は変更不要を原則とし、**追加ステップは `screenshots.spec.ts` のみ**で足りるか確認する。
